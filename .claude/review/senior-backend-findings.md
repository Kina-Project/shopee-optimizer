## 改善提案トップ3

### 1. 重い処理の非同期化とタスクキュー導入（優先度: 最高）

`process_stream` 内の同期ジェネレータは、FastAPIのイベントループをブロックする最大の問題。以下の改善を推奨:

- **短期:** `asyncio.to_thread()` または `run_in_executor()` で同期I/Oをスレッドプールに逃がす
- **中期:** Celery / Cloud Tasks / Pub/Sub 等のタスクキューを導入し、重い処理（動画生成、Drive/Sheetsアクセス）をワーカーに委譲。SSEはタスク進捗のポーリング/プッシュに限定する
- これにより、複数ユーザーの同時リクエスト時にサーバーがハングする問題を解消

### 2. インメモリ状態管理の永続化（優先度: 高）

`BATCH_STORE` (グローバル辞書) はプロセス再起動でファイルから復元されるが、複数ワーカー間での整合性がない:

- **短期:** SQLite or Cloud Firestore でバッチ状態を永続化
- **中期:** Redis をセッションストアとして導入し、ワーカー間で状態共有
- Cloud Run のスケーリング（複数インスタンス）を見据えた設計変更が必要

### 3. レスポンス標準化とPydanticモデルの活用（優先度: 中）

- 全エンドポイントで統一レスポンスエンベロープを定義: `{"ok": bool, "data": ..., "error": {"code": str, "message": str}}`
- リクエストボディに Pydantic `BaseModel` を適用し、FastAPIの自動バリデーション・OpenAPIドキュメント生成を活用
- エラーレスポンスは例外ハンドラで一元管理し、内部情報の漏洩を防止

## 潜在的な技術的負債

1. **フロントエンド埋め込み:** `INDEX_HTML` (約1000行のHTML/CSS/JS) がPythonファイルに直書き。今後のUI変更がバックエンドのデプロイを必要とし、フロントエンド専任者が触りにくい。SPAフレームワーク (React/Vue) への分離、または最低限 Jinja2テンプレートへの外出しが必要。

2. **テストの不在:** ユニットテスト・統合テストが確認できない。外部APIに依存するコアロジック (`fetch_amazon_product`, `upload_to_drive`, `write_to_spreadsheet`) のモック可能な設計になっていない（関数内で直接 `requests.get` や `gspread.authorize` を呼んでいる）。DI（依存性注入）パターンへの移行が望ましい。

3. **認証/認可の不在:** APIエンドポイントに認証がない。`/process-stream` を叩くだけで有料API（Rainforest, OpenAI, fal.ai）のクレジットを消費できる。Cloud Run IAM や API Key認証の導入が必要。

4. **ログ構造化の不足:** `logger.info/warning` は使われているが、構造化ログ（JSON形式）ではない。Cloud Loggingとの統合時にパース困難。リクエストID、ASIN、バッチIDなどのコンテキスト情報が一貫してログに含まれていない。

5. **ffmpeg依存のハードコード:** `subprocess.run(["ffmpeg", ...])` が多数あり、ffmpegのインストール有無やバージョン差異への対応がない。Dockerイメージでの管理は想定されているが、ローカル開発時のセットアップ手順が不明。

6. **Sheetsスキーマの暗黙的依存:** スプレッドシートの列順序がコードにハードコード（`r[0]`〜`r[13]`）。シートの列追加・順序変更でサイレントに壊れる。

## 次のPhaseへの引き継ぎ事項

1. **Cloud Run デプロイ時の注意:** `BATCH_STORE` のインメモリ管理は単一インスタンス前提。`--max-instances=1` の制約が外れた瞬間にデータ不整合が起こるため、永続化層の導入を最優先で実施すること。

2. **外部APIキーの管理:** Rainforest API Key, OpenAI API Key, FAL_KEY, Google CSE API Key, Drive OAuth credentials が環境変数で管理されているが、Secret Manager への移行を推奨。特に `SPREADSHEET_ID` のデフォルト値としてコード中に実IDがある点は早急に除去すべき。

3. **同時実行制御:** 動画生成（fal.ai）は1リクエストあたり30秒〜2分かかる。10件バッチの場合、SSE接続が10分以上維持される。Cloud Run のリクエストタイムアウト（デフォルト300秒）を考慮し、タイムアウト延長またはアーキテクチャ変更（非同期ジョブ+ポーリング）が必要。

4. **コスト管理:** Rainforest API（クレジット課金）、OpenAI GPT-4o（画像生成高額）、fal.ai（動画生成課金）を自由に呼び出せる状態。ユーザーごとの利用量制限、または承認フローの導入を検討すること。

5. **監視とアラート:** エラー率、外部API応答時間、動画生成成功率のメトリクス収集が未実装。Cloud Monitoring + Error Reporting の設定を次Phaseで対応。

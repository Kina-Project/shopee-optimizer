## 改善提案トップ3

### 提案1: フロントエンドコードの外部ファイル分離とコンポーネント化

- **問題：** 約980行のHTML/CSS/JSが Python 文字列リテラル内にハードコードされており、シンタックスハイライト・リンティング・型チェック・テストが一切効かない。履歴ページも含めると合計3ページ分のCSS重複がある。
- **解決策：** HTML/CSS/JS を `static/` ディレクトリに分離し、FastAPI の `StaticFiles` マウントで配信する。JS は ES Modules 形式でファイル分割し、ビルドツール（Vite等）を導入。共通スタイルは1つの CSS ファイルに統合。履歴ページは Jinja2 テンプレートエンジンで共通レイアウトを共有。
- **実装コスト：** 高
- **期待効果：** 開発効率の大幅向上、バグ検出の容易化、CSS重複排除によるページサイズ削減、チームでの並行開発が可能に。

### 提案2: アクセシビリティの基盤整備（label紐付け、ARIA属性、コントラスト修正）

- **問題：** フォーム要素にlabelがない、タブUIにARIA role/属性がない、プログレスバーにARIA属性がない、色のコントラスト比がWCAG AA基準を満たさない箇所がある。スクリーンリーダーユーザーには操作不可能に近い状態。
- **解決策：** (1) 全 `<input>` / `<select>` / `<textarea>` に `<label>` または `aria-label` を付与。(2) タブUIに `role="tablist"` / `role="tab"` / `aria-selected` を実装。(3) `--c-text-muted` を `#6b7280` 程度に変更しコントラスト比4.5:1以上を確保。(4) プログレスバーに `role="progressbar"` + `aria-valuenow` を付与。(5) ステップドットに色以外の識別手段（アイコン）を追加。
- **実装コスト：** 中
- **期待効果：** WCAG 2.1 AA準拠に近づき、スクリーンリーダー・キーボード操作ユーザーにも利用可能なツールになる。法的リスクの低減。

### 提案3: ユーザーフィードバックの統一化と非同期操作のUX改善

- **問題：** エラー通知が `alert()` 頼り、再生成・確定・画像追加等の非同期操作にローディング状態がない、`renderReview()` が動画再生を中断する。
- **解決策：** (1) `alert()` をインラインのトースト/メッセージコンポーネントに置換し、「次に何をすべきか」の情報を付与。(2) 全非同期ボタンにローディングスピナー + `disabled` 制御を追加。(3) `renderReview()` を差分更新方式に変更し、`<video>` 要素の再生状態を保持。(4) `JSON.parse` に `try-catch` を追加し、不正データでストリームが壊れないようにする。
- **実装コスト：** 中
- **期待効果：** ユーザーの操作迷子・二重送信を防止。エラー発生時の自己解決率向上。動画レビュー時の体験が大幅に改善。

## 次のPhaseへの引き継ぎ事項

1. **セキュリティ観点の深堀りが必要：** `innerHTML` + 文字列テンプレートによるDOM構築は、現状 `esc()` 関数でテキストノードのエスケープは行っているが、`onclick` ハンドラの属性コンテキストでのエスケープが不完全。ASINに特殊文字が含まれるケースでのインジェクションリスクを精査すべき。サーバーサイドHTML生成（履歴ページ）でも `html.escape` と `quote()` の使い分けを統一する必要がある。

2. **パフォーマンス計測の実施：** Google Fonts 3ファミリの読み込みがLCP/CLSに与える影響を Lighthouse 等で計測すべき。特に Noto Sans JP は数百KBになりうるため、`font-display: swap` の効果確認と、`unicode-range` によるサブセット化を検討。

3. **テスト戦略の策定：** 現状フロントエンドのユニットテスト・E2Eテストが一切ない（テスト可能な構造にもなっていない）。提案1のファイル分離後、Playwright 等でSSEストリーム処理・タブ切替・再生成フローの統合テストを整備すべき。

4. **状態管理の見直し：** グローバル変数（`reviewProducts`, `progressProducts`, `activeProductIndex` 等）による状態管理は、現在の規模でも追跡が困難になりつつある。ファイル分離と同時に、簡易的な状態管理パターン（Pub/Sub やシンプルなStore）の導入を検討。

5. **エラーリカバリ戦略：** SSE接続断時の自動再接続、バッチ処理の途中再開（サーバーサイドにバッチ状態は保存されている）、ネットワークエラー時のリトライUI等、プロダクション運用に必要なエラーリカバリ機構を設計すべき。

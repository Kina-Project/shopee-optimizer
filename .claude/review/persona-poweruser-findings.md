# ペルソナ「パワーユーザー（鈴木健太）」レビュー結果

## このペルソナが感じるフラストレーション

鈴木健太はスタートアップのマーケターとして、毎日数十件の商品ページを最適化する必要がある。このツールを使い込むほど、以下のフラストレーションが蓄積する。

1. **キーボードショートカットが一切ない**: URL入力、タブ切替、実行、レビュー操作すべてがマウスクリック前提。Ctrl+Enterで即実行、Tab/Shift+Tabでタブ移動、といった基本的なショートカットすら未実装。毎日使うツールとしてこれは致命的。

2. **CSVやJSONでのデータエクスポート機能がない**: 処理結果を他のツール（スプレッドシート分析、社内ダッシュボード）に流し込む手段がない。APIの`/batches`や`/batch/{id}`でJSONは取得できるが、UIからワンクリックでCSVダウンロードする導線がない。

3. **API連携は存在するが、ドキュメントもAPIキー認証もない**: FastAPIの自動ドキュメント（/docs）は使えるが、認証機構がゼロ。外部ツールからの自動化連携を組む際にセキュリティ上の不安が残る。APIキーやOAuthトークンでの認証が欲しい。

4. **一括操作が中途半端**: URLの一括入力はタブ方式で1件ずつ。テキストエリアに複数URLを改行区切りで貼り付けて一発投入、という最速パターンが使えない。10件が上限なのも、大量商品を扱うマーケターには物足りない。

5. **処理結果のフィルタリング・ソート・検索がない**: 履歴ページ（`/history`）は全件をHTML生成して返すだけ。ASIN検索、日付フィルタ、ステータスフィルタがない。100件超えると目視で探すしかない。

6. **バッチ処理がインメモリ（`BATCH_STORE`辞書）**: サーバー再起動でメモリ上のデータが消える。ファイル永続化（`save_batch_state`）はあるが、最大200件しかロードしない（L188）。大量運用時にデータロストのリスクがある。

## 「さすが、よくできている」と感じるポイント

1. **SSE（Server-Sent Events）によるリアルタイム進捗表示**: 処理の各ステップが即座にUIに反映される。ポーリングではなくストリーミングで実装しており、UX品質が高い。ストリーム無応答のウォッチドッグタイマー（15秒タイムアウト表示）もある（L797-803）。

2. **タブ型UIでの複数商品同時処理**: 入力・進捗・レビューの3フェーズそれぞれにタブUIがあり、複数商品の状態を把握しやすい。プログレスバーのパーセンテージ表示も直感的。

3. **動画の再生成ワークフロー**: 演出（effect）変更、画像選択、追加プロンプト指定での再生成が1画面で完結する。バージョン管理（v1, v2...）も自動で、過去バージョンとの比較が可能。

4. **画像不足時の自動検出と補完検索**: 3枚未満の場合に警告を表示し、Google画像検索で補完できるフローが組み込まれている。ユーザーが手動で画像を探す手間を省いている。

5. **バッチ状態のファイル永続化**: `save_batch_state()`でJSON保存しており、サーバー再起動後も`load_batch_store()`で復元される。完全なインメモリではない点は評価できる。

6. **パストラバーサル対策**: ファイル配信エンドポイント（`/files/`）で`is_relative_to()`チェックを実装（L1554）。セキュリティ意識がある。

7. **履歴からの「再開」機能**: 過去に処理した商品URLをワンクリックで入力フォームに戻せる`resume_url`パラメータ対応（L668-675）。

## パワーユーザーを引き止めるための改善TOP3

### 1. 一括URL入力とバッチ上限拡張（優先度スコア: 81）
**現状**: タブを1つずつ追加してURL入力。最大10件。
**改善案**: テキストエリアに改行区切りで複数URLを貼り付ける入力モードを追加。バッチ上限を50件以上に拡張（バックエンドの並列処理も検討）。これだけでパワーユーザーの入力工数が1/5以下になる。
**実装の手がかり**: `collectInputUrls()`（L662-665）を拡張し、テキストエリアからの一括パース機能を追加。`MAX_BATCH_SIZE`（L45）の引き上げ。

### 2. CSV/JSONエクスポート + キーボードショートカット（優先度スコア: 63+56）
**現状**: 処理結果のエクスポート手段がなく、全操作がマウス前提。
**改善案**:
- レビュー画面・履歴画面に「CSVダウンロード」「JSONダウンロード」ボタンを追加
- 主要操作にキーボードショートカットを割り当て（Ctrl+Enter: 実行、Ctrl+←/→: タブ移動、Ctrl+S: 確定、Escape: キャンセル）
- ショートカット一覧のヘルプモーダル（`?`キーで表示）

### 3. バックグラウンド処理とリトライ機構（優先度スコア: 72+42）
**現状**: ブラウザタブを閉じると進捗が消失。エラー時の個別リトライ不可。
**改善案**:
- 処理をサーバーサイドで完全非同期化し、ブラウザを閉じても継続。完了時にメール/Slack通知
- `batch_id`で後から進捗確認・レビュー画面に復帰できるようにする（`/batch/{id}`エンドポイントは既に存在するので、UIの復帰導線を追加するだけ）
- 失敗した商品の個別リトライボタンを進捗画面に追加

## 次のPhaseへの引き継ぎ事項

1. **HTMLがPython文字列リテラルに直接埋め込まれている（L198-1178、約1000行）**: フロントエンドの変更がバックエンドのデプロイを伴う。テンプレートエンジンまたはSPA分離を検討すべき。パワーユーザー向けの高度なUI（ドラッグ&ドロップ、キーボードナビゲーション等）を実装するにはこの構造が大きな障壁になる。

2. **認証・認可の設計が未着手**: 現状は誰でもAPIにアクセスできる。マルチテナント化やチーム利用を想定するなら、APIキー認証またはOAuth2の導入が必要。FastAPIの`Depends`を活用した認証ミドルウェア設計を次Phaseで検討。

3. **インメモリ`BATCH_STORE`のスケーラビリティ**: 現状はシングルインスタンス前提。Cloud Runのオートスケールで複数インスタンスが立つとバッチ状態が共有されない。Redis/Firestoreなどの外部ストアへの移行を検討。

4. **`/history`エンドポイントのパフォーマンス**: 毎回スプレッドシートAPIを叩いて最大1000件取得→HTML生成。キャッシュレイヤー（TTL付きメモリキャッシュまたはRedis）の導入が必要。パワーユーザーは履歴を頻繁に参照するため、レスポンスタイムが体感品質に直結する。

5. **フロントエンドのエラーハンドリング**: `regenerate()`（L1079-1098）や`finalizeBatch()`（L1161-1174）でAPIエラー時に`alert()`を使用。パワーユーザーにはトースト通知や非破壊的なインラインエラー表示が望ましい。

6. **Webhook/コールバック対応**: 外部システム（Slack、社内ツール）との連携のため、処理完了時にWebhook URLへPOSTする機能を追加すると、パワーユーザーのワークフロー自動化を大幅に支援できる。
